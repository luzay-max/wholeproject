# 数据字典业务迁移计划书

## 1. 迁移目标
将系统中的硬编码枚举值（如物品类型、审核状态、用户角色等）替换为基于数据库的动态数据字典，以提高系统的灵活性和可维护性。

## 2. 涉及范围

### 2.1 前端组件 (Frontend)
需要修改以下组件中的硬编码下拉列表 (`options` / `categories`)：
- `src/components/lostFind/LostForm.vue` (物品类型)
- `src/components/lostFind/FindForm.vue` (物品类型)
- `src/components/lostFind/InfoList.vue` (物品类型筛选, 状态筛选)
- `src/views/admin/UserManagement.vue` (如果存在，用户角色/状态筛选)

### 2.2 后端实体 (Backend)
涉及以下实体的字段：
- `LostInfo.java`: `type` (物品类型), `status` (审核状态)
- `FindInfo.java`: `type` (物品类型), `status` (审核状态)
- `User.java`: `role` (用户角色), `status` (用户状态)
- `HonorPeriod.java`: `periodType` (周期类型)

## 3. 字典定义 (Dictionary Schema)

我们需要在 `sys_dict_type` 和 `sys_dict_data` 中完善以下字典：

### 3.1 物品类型 (`item_type`) [已初始化]
- `ID_CARD`: 证件
- `ELECTRONICS`: 电子产品
- `BOOKS`: 书籍文具
- `CLOTHES`: 衣物
- `DAILY_NECESSITIES`: 生活用品
- `OTHER`: 其他

### 3.2 信息审核状态 (`info_status`) [新增]
用于 `LostInfo` 和 `FindInfo` 的 `status` 字段。
- `PENDING`: 待审核 (Warning)
- `APPROVED`: 已发布 (Success)
- `REJECTED`: 已拒绝 (Danger)
- `SOLVED`: 已解决 (Info)

### 3.3 用户角色 (`user_role`) [新增]
用于 `User` 的 `role` 字段。
- `STUDENT`: 学生
- `TEACHER`: 教师
- `ADMIN`: 管理员

### 3.4 用户状态 (`user_status`) [新增]
用于 `User` 的 `status` 字段。
- `0`: 正常 (Success)
- `1`: 封禁 (Danger)

## 4. 迁移步骤

### 第一阶段：数据准备
1.  更新 `sys_dict_init.sql`，补充 `info_status`, `user_role`, `user_status` 的初始化数据。
2.  执行 SQL 脚本，确保字典表中有完整数据。

### 第二阶段：前端重构
1.  **组件封装**：
    - 完善 `DictSelect` 组件，支持传入 `dictType` 自动加载选项。
    - 完善 `DictTag` 组件，用于在列表页回显字典 Label。
2.  **页面替换**：
    - 修改 `LostForm.vue` / `FindForm.vue`: 将 `<el-select>` 替换为 `<DictSelect dict-type="item_type" />`。
    - 修改 `InfoList.vue`: 筛选条件使用 `<DictSelect>`，列表列使用 `<DictTag>` 显示状态和类型。

### 第三阶段：后端与数据兼容
1.  **数据清洗**：
    - 检查现有的 `lost_info` 和 `find_info` 表，确保 `type` 和 `status` 字段的值与字典定义的 `dict_value` 一致。
    - 如果存在旧数据（如使用中文存储类型），需要执行 SQL 脚本进行批量更新（如 `UPDATE lost_info SET type='ID_CARD' WHERE type='证件'`）。

### 第四阶段：验证
1.  **功能验证**：
    - 发布流程：选择字典项是否正常保存。
    - 列表展示：字典值是否正确翻译为 Label。
    - 筛选功能：使用字典项筛选是否生效。
